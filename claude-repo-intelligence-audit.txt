====================================================================
1) Executive Snapshot
====================================================================
This repository is a multi-surface Christian education platform with two primary delivery modes:

A) Root static platform (main public experience)
- Large HTML/CSS/JS site with studies, devotions, games, videos, and newsroom.
- Firebase-hosted, JSON-driven content updates, many standalone pages.

B) Nested modern app in christianity-of-christ
- Next.js + TypeScript course app with API routes, Prisma models, tests, and build/export workflow.
- Exported static output is copied to systematic-reconstruction for deployment.

Operationally, the root static site is broad and active, while the nested Next.js app represents a structured modernization path with some production-ready parts and some intentional stubs.

====================================================================
2) Architecture and Delivery Model
====================================================================
2.1 Root site architecture
- Entry: index.html
- Main behavior scripts:
  - Indexstyles.js (auth/profile/account-mode logic + Firebase SDK usage)
  - nbtt-features.js (search index, bookmarks, theme behavior, reading progress UX)
  - script.js (homepage commandment/video card rendering, quiz modal behavior)
  - ledger.js (Evidence Desk/newsroom card feed, post rendering, trending drawer)
- Data-first content pattern: newsroom.json, commandments.json, script.json, biblestories.json, ledgerData.json.
- Site composition: many static route pages at root and in Studies, Devotions, games.

2.2 Next.js sub-application architecture
- Folder: christianity-of-christ
- App Router structure in src/app, plus API routes under src/app/api.
- Data layer via Prisma (SQLite default), adapter concept for PostgreSQL.
- Test layer via Vitest.
- Content layer via MDX modules under content/modules.
- Build artifacts produced in out and copied into systematic-reconstruction by root scripts.

2.3 Deployment architecture
- Firebase Hosting serves from repository root (public is ".").
- GitHub Actions workflows deploy preview/live hosting channels.
- Root scripts orchestrate nested app build + export sync + firebase deploy.

14) Claude Guidance: How to Match Repo Style Correctly
====================================================================
If Claude is editing this repo, it should:

1) Preserve hybrid architecture assumptions.
- Do not treat repo as only Next.js or only static.
- Respect root static site as primary live surface.

2) Preserve metadata and SEO discipline on content pages.
- Keep canonical, OG, Twitter, JSON-LD patterns consistent.

3) Preserve theological/editorial structure for newsroom and devotional content.

4) Prefer data-driven updates when content already flows from JSON.
- Update source JSON and rendering assumptions together.

5) Avoid breaking Firebase integration assumptions.
- Keep auth/profile/leaderboard and rules compatibility in mind.

6) Treat some modules as intentionally stubbed.
- Expand stubs with explicit transition-safe steps rather than replacing architecture abruptly.

7) Keep visual language aligned with existing design choices.
